# Novel Scene

把已批准的大纲 beat 或场景 brief 转成可验证的场景契约。先确认场景为什么必须存在，再设计读者如何看到事件；不要用漂亮对白或正文片段掩盖因果和连续性缺口。

## 核心契约

- 在任何小说级读取或写入前要求用户明确给出并确认 `novel_id`。缺失时转交 `story-os-manager`，不要用 `active_novel_id` 或最近文件代替选择。
- 把 `EVT-*` 视为世界内实际发生的事件，把 `SCN-*` 视为读者看到事件的叙事呈现。始终分开 `story_time` 与 `narrative_order`。
- 把版本化源文件视为唯一事实源。区分 Canon、proposal、research 和 manuscript；只允许作者批准 Canon。
- 只读取当前小说和 manifest 中显式声明、带版本的系列或共享世界依赖。不得把其他小说当作隐式参考。
- 在读写前声明任务、场景范围、上下文清单、上下文预算、修改文件和当前门禁。
- 把新场景、事件和状态变化保持为 `proposed`。作者批准本次设计只允许写入提案，不自动晋升 Canon。
- 使用 `templates/narrative/scene-contract.yaml` 与必要的 `templates/entities/event.yaml`；复制后填写，不编辑模板本身。
- 把正式章节写作交给 `novel-chapter`，把大纲结构变化交给 `novel-outline`，把 Canon 变化交给对应专用流程。

## 1. 绑定场景并先验证

1. 读取 `workspace/manifest.yaml`，验证用户给出的 `novel_id` 已注册且路径存在。
2. 读取目标小说 manifest，核对 `novel_id`、当前卷章场景、类型承诺、依赖和可见性。
3. 确认任务是新建、修订、拆分、合并、换 POV、重排还是只读审查。
4. 确认唯一目标：已批准的大纲 beat、章节计划或现有 `SCN-*`；没有上层依赖时先转交 `novel-outline`。
5. 检查工作树是否有与目标场景或事件重叠的用户改动；有重叠时停止并报告。
6. 从仓库根目录运行 `npm run validate:story`；Windows PowerShell 执行策略阻止脚本入口时使用 `npm.cmd run validate:story`。
7. 报告全部验证错误。Schema、owner、引用、时间、知识、物品或共享世界错误会影响本场时，停止设计，不顺手修复。

输出启动契约：`novel_id`、任务类型、目标 beat/章节/场景、验证结果、拟读取文件、拟修改文件、上下文预算和第一个作者门禁。

## 2. 只加载最小上下文

默认只加载：

- 目标小说 manifest、相关大纲 beat 和本章计划。
- 上一场 `exit_state`、下一场已确定的入口依赖，以及目标现有场景契约。
- POV 人物、实际在场人物、主要地点和直接生效的世界规则。
- 本场引用或改变的 `EVT-*`、`FACT-*`、`KNOW-*`、`REL-*`、`ITEM-*`、`THREAD-*` 和 `FORESH-*`。
- 与本场直接相关的最新作者决定和显式共享世界版本。

使用 `rg` 按 ID 与路径精确定位。维护上下文清单，列出已读取文件、实体 ID、依赖版本和明确排除项。不要默认读取整本大纲、整部正文、其他小说、根目录历史计划、构建物、依赖目录或日志。需要扩大预算时，先说明新增内容、原因和数量，再请求许可。

## 3. 确认场景职责、POV 与时空

先用一句因果句定义职责：

```text
因为 <前置压力>，<POV 人物> 为了 <具体目标> 选择 <行动>，
结果 <关键状态变化>，迫使 <下一步行动或问题>。
```

同时确认：

- 本场推进主线、人物弧、关系、谜团、世界规则、伏笔或类型体验中的哪些项。
- 删除本场会失去什么不可替代的因果或读者体验。
- 为什么此刻发生，为什么由这个 POV 呈现；POV 应拥有最有代价的选择和受限的信息，不一定是知道最多的人。
- `narrative_order` 在哪一卷、章、场；相关 `EVT-*` 的 `story_time`、精度和地点是什么。
- 倒叙、梦境、传闻、伪造记录或不可靠叙述是否需要显式标记。
- 多地点、多时段或多 POV 是否仍属于一条连续行动；若需要跳跃且入口状态重置，拆成多个场景。

### Gate A：职责与视角批准

要求作者确认场景职责、POV、叙事位置、故事时空和拆场判断。未确认时不要设计详细节拍或创建 `SCN-*` 文件。

## 4. 运行连续性预检

从上一场出口和相关事件重建 `entry_state`：

- 故事时间、人物所在地点、移动时间与可达性。
- 身体、伤势、疲劳、精神、能力消耗、金钱和关键资源。
- 每件关键物品的持有人、位置和状态。
- POV 与在场人物已知、怀疑、误信和未知的事实。
- 方向性关系、承诺、秘密、义务、恐惧与权力状态。
- 尚未兑现的动作、威胁、期限和上一场 `exit_hook`。
- 当前生效的世界规则、例外、代价和类型承诺。

把本场会主动依据的知识列为 `knowledge_used_ids`。逐项确认对应 `KNOW-*` 属于正确人物，且 `learned_in_scene_id` 或 `learned_at_event_id` 不晚于本场；不要根据作者层真相让人物提前行动。

发现冲突时输出 ID、来源文件、影响和最小解阻动作，不自动选择方便的版本。

### Gate B：入口状态批准

要求作者确认 `entry_state`、知识上限、物品状态和连续性风险。阻塞冲突未解决时不要进入节拍方案。

## 5. 选择 Scene/Sequel 模式

先选择本场主模式，不机械套满两组框架：

- **Scene（行动）**：Goal → Conflict → Disaster。把具体目标写入 `goal`，升级阻力写入 `conflict/turning_point`，把失败、代价或更坏局面写入 `resolution/exit_hook`。
- **Sequel（反应）**：Reaction → Dilemma → Decision。把即时反应与承接状态写入 `entry_state`，困境写入 `crisis`，决定写入 `climax`，把新目标写入 `resolution/exit_hook`。
- **混合**：只有当反应确实导向新的行动循环时使用；不要为了填字段把一场塞成两场。

若实现路径尚未确定，提供两到三个节拍方案；每个方案说明读者体验、人物代价、信息揭示、连续性风险和下一场压力。若场景解法已被作者锁定，只比较危机选择或转折强度，不无谓扩展范围。

### Gate C：模式与方案选择

要求作者选择模式、节拍方案或危机方向。未选择时不要替作者定案，也不要写入场景契约。

## 6. 建立场景因果链

按下列顺序填写，确保每一项改变下一项的条件：

```text
entry_state
  -> goal
  -> conflict
  -> turning_point
  -> crisis
  -> climax
  -> resolution
  -> exit_state
  -> exit_hook
```

- `goal`：POV 此刻想取得的具体、可观察结果，不写抽象主题。
- `conflict`：有行动能力的对手、环境、规则或内部压力，以及它为何此刻生效。
- `turning_point`：让原计划失效、改变风险或重新定义目标的新事实、行动或代价。
- `crisis`：两个都带真实代价的选择，并写清拖延或拒绝选择的后果。
- `climax`：POV 基于 Want、Need、Wound、Lie、价值观或压力采取的不可撤销行动。
- `resolution`：高潮行动直接造成的成功、失败、代价和价值变化。
- `exit_state`：精确列出地点、时间、伤势、资源、物品、知识、关系、承诺和风险的新状态。
- `exit_hook`：由本场结果产生、会迫使下一场行动的具体问题、威胁、承诺或决定；不要用无因果的新消息截断章节。

用 Five Commandments 复核重要场景：Inciting Incident 对应触发压力，Progressive Complication / Turning Point 对应升级与转折，Crisis 对应代价选择，Climax 对应人物行动，Resolution 对应结果。五项必须属于同一因果链，但不要求每个短过渡场景拥有相同体量。

## 7. 记录事件与状态变化

### 因果事件

- 复用已存在且语义相同的 `EVT-*`；不要为同一世界事实创建重复事件。
- 新世界内行动需要独立引用时，复制 event 模板并保持 `status: proposed`。
- 填写 `story_time`、`participants`、`location_id`、`caused_by_event_ids`、`consequence_event_ids` 和 `facts_established`。
- 用事件 ID 表达因果，不用时间相邻或叙事相邻代替原因。
- 一个场景可呈现多个事件，一个事件也可被多个场景从不同视角呈现。

### 物品变化

把关键物品转移写入对应事件的结构化 `item_changes`，不要只藏在 summary 或正文设想中。每条包含：

```text
item_id
from_holder_id / to_holder_id
from_location_id / to_location_id
state_after
```

允许无持有人或未知地点时使用 Schema 允许的 `null`。核对转移前状态与上一事件一致；同一故事时点不得把同一物品交给互斥持有人。

### 知识、关系与伏笔

- `knowledge_used_ids`：本场行动实际依赖、且入场前已可用的 `KNOW-*`。
- `knowledge_changes`：本场新增、修正或推翻的知识状态；对应记录必须指向来源事件/场景。
- `relationship_changes`：只引用本场实际生效或结束的方向性 `REL-*`，并用相关事件解释原因。
- `foreshadowing_ids`：只引用本场种植、推进、回收或放弃的 `FORESH-*`；核对读者可见信息与真实含义没有提前泄露。
- 新事实、关系、知识或伏笔实体都保持 proposal；需要改 Canon 时转交变更流程。

伤势、资源和精神变化当前没有独立场景字段时，明确写入事件 summary 与 `exit_state`，并在后续结构化实体可用时创建提案；不要添加 narrative Schema 禁止的临时字段。

### Gate D：状态变化批准

要求作者确认事件因果、事实、`item_changes`、`knowledge_used_ids`、知识/关系变化、伏笔和 `exit_state`。未确认时不要落盘。

## 8. 执行场景压力测试

写入前逐项检查：

1. **删除**：删掉本场后，下一场是否仍能无损发生？若能，合并、压缩或删除；若不能，写明具体依赖。
2. **换序**：与相邻场景互换后，时间、知识、物品、关系或因果哪里首先断裂？
3. **换 POV**：哪个人物承担最大选择代价并拥有恰当的信息限制？换人是否更强或造成不该知道的信息泄漏？
4. **拆分/合并**：跨地点、跨时间、目标重置或价值变化两次时考虑拆分；职责重复时考虑合并。
5. **因果**：相邻节拍能否用“因为 / 所以”连接？若只能用“然后”，补齐触发、选择或后果。
6. **巧合**：巧合可以制造麻烦，不能替人物解决核心问题。
7. **升级**：冲突是否真正改变计划、代价或选择，而非只增加噪音？
8. **类型承诺**：本场是否兑现、推进或有意识延迟读者承诺，且延迟有回报计划？
9. **出口**：`exit_state` 能否直接成为下一场 `entry_state`；`exit_hook` 是否由本场结果产生？

输出保留、修改、合并、拆分或删除建议，并说明影响 ID。压力测试是诊断，不自动改上层大纲。

### Gate E：契约与写入批准

向作者展示完整契约、压力测试结论、未决问题、目标文件和必要事件提案。只有作者明确批准设计与写入范围后才落盘。

## 9. 写入 proposal 并验证

获得 Gate E 批准后：

1. 使用作者或项目管理流程分配的未占用永久 `SCN-*` / `EVT-*` ID；没有已批准 ID 时停止，不自行复用占位 ID。
2. 复制 `templates/narrative/scene-contract.yaml` 到目标小说叙事目录，立即替换所有占位值。
3. 保持场景 `status: proposed`、`visibility: internal`，填写真实 `owner.novel_id`、叙事顺序、引用和状态变化。
4. 当人物在本场依据既有知识行动时，显式加入 Schema 允许的 `knowledge_used_ids`；即使当前模板未展示该可选字段，也不要省略必要知识依赖。
5. 为必要的新事件复制 `templates/entities/event.yaml`，保持 `status: proposed`；有物品转移时按 narrative Schema 加入 `item_changes`。
6. 不写章节正文、对白试写或场景散文，不静默改变大纲、人物历史、世界规则或共享世界。
7. 解析 YAML，并再次运行 `npm run validate:story`。记录命令、结果和 warning；验证失败时不得宣称完成。
8. 检查 Git diff，确认只修改授权文件且没有跨小说泄漏。除非用户明确要求，否则不要提交 Git。

## 输出格式

每次响应至少包含：

- 绑定的 `novel_id`、目标场景和当前 Gate。
- 验证结果、上下文清单和使用的实体 ID / 共享世界版本。
- 场景职责、POV、时空、Scene/Sequel 模式和因果链。
- `entry_state`、`exit_state`、`exit_hook` 与状态变化摘要。
- 事件、物品、知识、关系和伏笔引用。
- 压力测试结论、作者已确认决定、未批准内容、风险与阻塞问题。
- 精确停止点、下一步门禁、写入文件和验证证据；未获写入批准时明确写“未落盘”。

## 必须停止的情况

- 缺少明确 `novel_id`，或注册表、路径、manifest、owner 不一致。
- 缺少已批准的大纲依赖，或请求本 Skill 静默改变上层结构。
- 验证失败会影响本场，或目标场景/事件有重叠的未提交用户改动。
- 连续性预检发现时间、地点、知识、物品、关系或规则冲突尚未解决。
- 需要读取未声明的其他小说或未锁定版本的共享世界。
- 必须新增或修改 Canon 才能继续，但没有作者批准的变更流程。
- 当前 Gate 尚未获得作者确认，或 POV、危机选择、状态变化存在阻塞歧义。
- 请求开始写正式章节、越过场景范围、伪造 ID 或直接修改共享世界。

停止时报告具体文件、ID、失败门禁和最小解阻动作；不要自行选择最方便的故事版本。
