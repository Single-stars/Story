# Novel Character

把人物设计成会主动选择、付出代价并改变关系与情节状态的人。不要用本 Skill 替代 premise、大纲、场景、正文或 Canon 变更流程。

## 核心契约

- 在任何小说级读取或写入前要求用户明确给出并确认 `novel_id`；不得用 `active_novel_id`、唯一候选或最近编辑文件代替选择。
- 把版本化源文件视为唯一事实源，区分 Canon、proposal、research 和 manuscript。所有新人物、人物历史、秘密、关系和知识状态默认保持 `proposed`。
- 只允许作者批准 Canon。作者批准人物设计方向不等于批准其中每条新事实，也不等于授权直接修改已有 Canon。
- 只读取当前小说和 manifest 显式引用、带版本的系列或共享世界依赖；不得借用其他小说的人物、设定或反馈。
- 写入前声明目标文件、允许改动和门禁；预计超过 3 个文件时拆成独立批次。
- 发现需要改变已批准事实、共享世界人物或高影响人物历史时，转交 `novel-canon-change`，不要就地 retcon。

## 1. 绑定范围并先验证

1. 按 `story-os-manager` 读取工作区注册表，验证用户给出的 `novel_id`、目录和小说 manifest 一致。
2. 确认任务模式：新建人物、深化人物、人物审计、关系设计、知识边界审计或人物弧修订。
3. 检查工作树是否有与目标人物/关系/知识文件重叠的用户改动；存在重叠时停止。
4. 从仓库根目录运行 `npm run validate:story`；Windows PowerShell 入口受限时使用 `npm.cmd run validate:story`。
5. 报告与当前人物任务有关的 Schema、归属、引用、知识或连续性错误；不要顺手修复。
6. 回显唯一修改范围和第一个作者门禁。

启动输出至少包含：`novel_id`、任务模式、目标人物或临时标签、输入状态、验证结果、上下文清单、上下文预算、拟修改文件和阻塞问题。

## 2. 只加载最小上下文

默认只读取：

- `AGENTS.md`、工作区 manifest、目标小说 manifest。
- 已获作者选择的 premise，以及当前人物直接参与的 outline 层级或因果 beat。
- 目标人物现有 `CHAR-*`；与其直接相连的 `REL-*`、`KNOW-*` 和被引用的 `FACT-*`。
- 直接限制人物行动的规则、伤势、物品、事件、场景与作者决定。
- `templates/entities/character.yaml`、`templates/entities/relationship.yaml` 和 `schemas/canon-entity.schema.json`。
- 需要写知识记录时读取 `templates/entities/knowledge.yaml`；若模板不存在，只做对话级知识 proposal，停止知识文件写入并报告缺口。

使用 `rg` 按 ID、名字和路径精确定位。默认最多加载 8 个任务直接相关源文件；超出前说明新增文件、理由和数量。不要读取其他小说、整部正文、根目录历史计划、构建物、依赖目录或日志。

把输入分为：

- `canon_constraints`：已批准且本轮不能静默改变的事实。
- `approved_structure`：作者已选择的 premise、大纲职责和类型承诺。
- `existing_proposals`：可比较、可修改但未批准的候选。
- `research_limits`：未核实的现实、文化、职业、医疗或敏感议题。
- `open_questions`：缺失但不应由 AI 自行补齐的决定。

### Gate A：输入与职责确认

向作者确认任务模式、人物在故事中的职责、允许修改的既有内容和阻塞问题。缺少明确 `novel_id`、premise/outline 方向冲突或人物职责会改变故事核心时，停在此 Gate。

## 3. 定义角色职责而非剧情工具

先写一份 proposal 级角色职责：

- `narrative_job`：该人物在因果链中必须主动完成什么，而非“提供信息”“陪伴主角”等被动标签。
- `plot_pressure`：该人物如何制造、升级或改变选择条件。
- `relationship_job`：其欲望如何改变至少一段双向但不对称的关系。
- `theme_lens`：其选择如何检验主题问题，而不是代作者发言。
- `genre_delivery`：其行动兑现哪项读者承诺或必要体验。
- `independent_stake`：即使主角不在场，该人物仍会追求或害怕什么。
- `replaceability_test`：用无名信使、恋人、导师或反派手下替换后，哪些因果会失效；若没有，角色尚未成立。

配角可以承担较窄职责，但仍应有自己的目标、边界和一次有后果的选择。不要先用职业、身份标签或创伤替代人格。人物职责是作者侧的因果设计，不是人物自觉，也不是对白任务；禁止让角色用一句漂亮话说明自己“负责观察、负责怀疑或负责保护”。

## 4. 建立人物动力契约

为主要人物建立以下因果链；次要人物按任务需要缩减，但不得省略目标与限制。

### Want / Need / Wound / Lie

- `want`：人物主动追求、场景中可观察且能判断成败的外部目标。
- `need`：人物必须理解、接受或改变的内部需要；不能只是更抽象的 Want。
- `wound`：形成防御方式和选择模式的经历；说明影响，不用创伤强度替代深度。
- `lie`：人物当前相信且会持续制造代价的错误信念；必须能被行动证伪或强化。
- `pressure_link`：外部事件如何反复攻击 Wound/Lie，使 Want 与 Need 发生冲突。

用“因为 Wound，所以相信 Lie，因此追求 Want 并回避 Need；压力迫使其选择”检查链条。任一环只能靠作者解释、不能从选择中观察时，标记为弱连接。

### 三层冲突

- `external_conflict`：人物与对手、制度、环境、规则或资源限制争夺什么。
- `internal_conflict`：两种价值、欲望、恐惧或身份要求如何互斥。
- `interpersonal_conflict`：两名具体人物各自合理的目标为何不能同时满足。

三层冲突至少有两层互相加压。不要把“缺少沟通”当作唯一的人际冲突，也不要让内部冲突在关键行动中消失。

## 5. 检查能动性、能力与代价

### 能动性

为人物至少标出：

1. 一个主动目标。
2. 一个拒绝默认路线的选择。
3. 一个改变资源、信息、关系、风险或后续目标的行动。
4. 一个由本人承担、不能转嫁给旁人的后果。

主要人物再检查开局、中段与高潮压力下的选择是否逐级不可逆。巧合可以制造麻烦，不能替人物解决核心问题；被营救不等于无能动性，但人物必须对营救前后的选择条件产生作用。

### 能力边界

对每项能力记录用途、可靠条件、失败条件、学习或来源证据，以及资源、身体、社会、道德或关系代价。写入人物模板时把能力映射到 `capabilities`，把限制与代价映射到 `limits`；`limits` 至少一项。

若能力能在关键场面稳定解决问题，却没有成本、反制、盲区或失败后果，判定为“万能角色”风险。不要临场新增专长来解围。

## 6. 设计声音指纹

使用人物模板的现有字段：

- `voice.vocabulary`：常用词、职业词、口头禅和拒绝使用的标签。
- `voice.syntax`：句长、停顿、语序、修辞和情绪失控时的变化。
- `voice.notices_first`：进入场景时最先注意什么，以及价值观形成的感知过滤器。
- `voice.avoids`：回避的话题、称谓、情绪和表达方式。

补做三项对照：同一坏消息下如何说、向不同权力位置的人如何说、撒谎时哪个模式会变化。声音差异应来自经历、目标和关系，不靠方言拼写、口癖堆叠、格言式台词或每句都展示人设。若对白不像正常人在该压力下说出，优先降低“辨识度”而保留自然性。

## 7. 管理秘密与知识边界

把“作者知道”“读者已见”“人物相信”和“世界内为真”分开。人物不能依据作者层真相行动。

对每项关键秘密记录：持有人、关联 `FACT-*`、隐藏对象、隐瞒动机、泄露条件、错误认知和泄露代价。写入 `secrets` 时保持 `visibility` 与剧透级别一致。

需要正式知识记录时，每条 `KNOW-*` 必须包含：

- `character_id` 与 `fact_id`。
- `knowledge_state`：`unknown | suspects | believes_true | believes_false | verified`。
- `learned_at_event_id`、`learned_in_scene_id` 和 `source_character_id`；未知或不适用时使用 Schema 允许的 `null`。
- `confidence`：0 到 1，并与状态和来源质量相符。

没有现存 `FACT-*`、来源事件/场景尚未分配或知识模板缺失时，不伪造 ID；把它保持为 proposal 并列出解除条件。重点审查人物是否提前知道秘密、忘记已验证信息，或把错误认知写成客观事实。

## 8. 建立方向性关系

关系必须分别记录 `A -> B` 与 `B -> A`；不得假设信任、好感、恐惧、义务或筹码对称。

为每个方向记录：

- `dimensions.trust / affection / fear / obligation / leverage`，范围 0–1；数值不是道德评分。
- `public_label` 与 `private_reality`，后者写清秘密、误解、权力差和行动影响。
- `valid_from_event_id`、`valid_to_event_id` 与真正造成变化的 `caused_by_event_ids`。
- 当前关系怎样改变人物下一步可选行动，而不仅是“关系变好/变坏”。

重大阶段变化应关闭旧有效区间并创建新的 `REL-*` proposal；不要静默重写历史区间。尚无有效 `EVT-*` 时先保留关系变化提案，不制造占位 ID。

## 9. 进行人物弧压力测试

建立弧线账本：起始状态、Want/Need、Wound/Lie 防御、升级压力、关键选择、最低点、高潮选择、成功/失败代价和结束状态。与已批准 outline beat 对齐，但不要在本 Skill 中重写大纲。

逐项压力测试：

- **因果：**变化是否由本人选择与后果造成，而非旁白宣布成长。
- **升级：**后一次压力是否攻击更深的 Lie、关系或代价，而非重复同一争吵。
- **一致：**行为是否能回到目标、价值、创伤、知识或当时压力；意外行为是否有提前证据。
- **高潮：**最终选择是否要求放弃旧防御或付出成功代价。
- **余波：**弧线结束是否改变关系、能力使用、社会位置或行动方式。
- **删减：**删除该人物或关键选择后，主线是否几乎不变；若是，职责或交叉因果不足。

### Gate B：人物动力与弧线确认

提交职责、动力链、三层冲突、能动性节点、能力边界、声音指纹、秘密/知识边界和弧线压力测试。要求作者确认、选择变体或缩小范围；未确认时不要创建实体文件。

## 10. 审计失效风险

给每项风险提供具体证据、发生位置、影响和最小修改方向：

- **刻板印象：**身份只剩单一特征；群体标签替代个人欲望；文化或创伤只作异域感、反派证明或成长燃料。
- **工具人：**只提供线索、牺牲、安慰、背叛或爱情奖励；没有独立目标；换成无名功能角色也不影响因果。
- **万能角色：**能力总能命中；代价不持续；知道不可能知道的信息；所有人无理由信任；作者临场赋予解法。
- **被动主角：**关键节点由他人决定，主角只接受任务、获救或听取真相。
- **声音同质：**不同人物使用相同词汇、观察顺序和回避方式，仅靠口癖区分。
- **知识越权：**人物依据未学习的事实行动，或把 `believes_false` 当成已验证真相。
- **关系扁平：**双向记录数值相同，公开标签等同私人现实，关系变化不改变行动条件。
- **弧线失效：**高潮成功无需触碰 Lie，成长没有代价，或反派只为证明主角正确。

涉及现实身份、文化、医疗、创伤或歧视时，列出研究问题与停止条件；资料不足时转交 `novel-research`，不要用善意猜测填补。

### Gate C：网络、风险与写入确认

展示方向性关系图、知识边界、风险清单、拟创建/修改的实体和影响 ID。只有作者明确批准目标文件与 proposal 内容后才写入；确认设计不等于批准 Canon。

## 11. 写入 proposal

获得 Gate C 批准后：

1. 复制 `templates/entities/character.yaml` 到当前小说的 `canon/characters/`，立即替换占位值与未使用的永久 `CHAR-*`。
2. 需要关系文件时复制 `templates/entities/relationship.yaml`；每个方向一个 `REL-*`，保持有效事件区间和原因引用合法。
3. 需要知识文件时复制 `templates/entities/knowledge.yaml`。若模板不存在，停止该部分写入，不得自建替代模板或绕过 Schema。
4. 保持 `status: proposed`、`visibility: internal`，除非作者对可见性另有明确授权；作者批准前不得改为 `canon`。
5. 只填写 Schema 允许的字段。扩展分析留在获批的设计提案或决策记录，不向实体文件添加自定义键。
6. 不伪造 `FACT/EVT/SCN/DEC` 引用、来源、批准时间或验证证据；未分配引用保持 Schema 允许的 `null`/空数组，或停止写入。
7. 高影响新历史、retcon 或共享世界变化转交 `novel-canon-change`。

写入后解析 YAML，运行 `npm run validate:story`，检查 owner、ID、引用、状态和可见性，并检查 Git diff 仅包含授权文件。除非用户明确要求，否则不要提交 Git。

## 输出格式

每次交付至少包含：

- `novel_id`、任务模式、当前 Gate、目标人物与范围。
- 验证结果、上下文清单与使用的 Canon/proposal/research ID。
- `role_contract`、`want_need_wound_lie`、三层冲突与能动性节点。
- 能力/限制/代价、声音指纹、秘密与知识边界。
- 两个方向的关系状态与弧线压力测试。
- 刻板印象、工具人、万能角色及连续性风险。
- 作者已确认决定、仍为 proposal 的内容和 `author_choice_required`。
- 精确 `stop_point`、下一门禁、建议路由、写入文件与验证证据。

## 必须停止的情况

- 缺少明确 `novel_id`，或注册表、路径、manifest、owner 不一致。
- premise/outline 职责未获选择，或本任务会静默改变故事核心。
- 需要读取未声明的其他小说或未锁定版本的共享世界。
- 目标文件有重叠的未提交改动。
- 新人物历史、秘密或关系会改变 Canon，但没有作者批准的变更流程。
- 知识记录缺少 `FACT-*`、来源 ID 或知识模板。
- 关系变化缺少有效事件 ID，或要求把方向性关系压成单一对称记录。
- 敏感事实缺少研究，且错误假设会改变人物基础。
- 当前 Gate 未获作者确认、上下文预算需无边界扩张或验证失败。
- 请求越权写大纲、场景、正文、共享世界或直接晋升 Canon。

停止时报告具体文件、ID、失败门禁和最小解阻动作。把后续结构调整路由到 `novel-outline`，场景落地路由到 `novel-scene`，研究路由到 `novel-research`，Canon/retcon 路由到 `novel-canon-change`。

## 完成检查

- 已显式绑定正确 `novel_id`，且上下文未跨小说泄漏。
- 角色职责、Want/Need/Wound/Lie 与三层冲突形成可观察因果。
- 主要人物在关键节点有主动选择、真实限制和持续代价。
- 声音指纹可区分且不依赖夸张口癖或身份刻板印象。
- 秘密、人物知识、读者知识和客观事实分层清楚。
- 关系按方向和事件区间记录，变化会影响行动条件。
- 人物弧通过因果、升级、高潮代价和删减测试。
- 已审计刻板、工具人、万能、被动、知识越权和关系扁平风险。
- 所有落盘实体保持 `proposed`，作者未批准内容没有晋升 Canon。
- YAML、Schema、引用、归属与 `npm run validate:story` 通过；已记录精确停点且未提交 Git。
