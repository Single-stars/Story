# Novel Line Edit

在句子和段落层提高可读性、精确度、节奏与声音一致性。不要把 line edit 变成剧情重写，也不要把编辑个人口味变成作品规则。

## 核心契约

- 先按 `story-os-manager` 绑定明确 `novel_id`、`CHAPTER-*`、唯一章节文件和精确段落/场景范围。
- 只在结构、人物与连续性审查完成后进入行文精修。存在结构或连续性 blocker 时停止并路由，不用漂亮句子遮住问题。
- 把情节事实、事件顺序、人物动机、关系状态、知识边界、世界规则、POV、叙述距离、时态和作者 voice 视为保护项。
- 默认先诊断，标出问题、证据和建议编辑强度；未完成诊断和作者强度确认前不改文件。
- 保留原稿可比较。默认在对话中给出“原文 / 建议稿 / 理由”；获得文件修改授权后使用最小补丁和 Git diff，不擅自创建 `.bak` 或覆盖历史。
- 不把“更华丽”“更文学”“更短”视为普遍正确。把可验证问题、作品契约和可选偏好分开。
- 新事实、能力、人物历史、隐喻设定或动机解释保持 proposal，并转交相应流程；不得因句子写得顺就晋升 Canon。

## 1. 绑定章节与编辑范围

1. 核对工作区注册、小说目录、manifest、章节 frontmatter 的 `novel_id` 一致。
2. 确认 `chapter_id`、卷章位置、文件路径、章节状态和编辑范围；范围可以是整章、指定场景、段落或行号。
3. 确认任务是诊断、轻改、标准精修、重点段落重写或作者反馈复核。
4. 检查目标文件是否有重叠的未提交用户改动；不得覆盖。
5. 运行 `npm run validate:story`；Windows PowerShell 入口受限时使用 `npm.cmd run validate:story`。
6. 读取最新 `novel-continuity` 结果；没有现成报告时对目标范围做最小连续性预检。

启动输出至少包含：`novel_id`、`chapter_id`、卷章位置、精确范围、章节状态、任务模式、验证结果、上下文预算、保护项和第一个作者门禁。

## 2. 只加载最小上下文

默认只读取：

- `AGENTS.md`、目标小说 manifest、目标章节的 frontmatter 与授权正文范围。
- 本范围直接对应的场景契约；必要时读取上一段/场出口和下一段/场入口。
- POV 人物的声音档案、知识边界、当前伤势/资源与直接关系。
- 本范围实际使用的事实、规则、物品、伏笔和 continuity 问题。
- 作者明确给出的风格偏好、禁改项和已批准措辞。

默认最多加载 8 个任务直接相关源文件或实体。使用 `rg` 按 `CHAPTER/SCN/CHAR/FACT/RULE` 和文本片段精确定位；需要扩大时先说明新增文件、理由和数量。不要读取其他小说或无关章节模仿文风，也不要用同类畅销书的表达替换作者 voice。

### Gate A：范围、保护项与上游门禁

确认以下内容后再诊断：

- 章节结构、人物动机和场景结果不在本轮改动范围。
- continuity 没有会使局部润色失去意义的 blocker。
- POV、人称、时态、叙述距离和作品语域已明确。
- 目标章节不是未授权修改的 `locked/published` 状态。

缺少任一项时报告具体阻塞并停止。结构交 `novel-developmental-edit`/`novel-outline`，人物交 `novel-character`，连续性交 `novel-continuity`，正文事实或场景结果交 `novel-chapter`/`novel-scene`。

## 3. 建立保真账本

在改写前列出 protected spans 与 protected relations：

### 不得漂移的内容

- 人名、称谓、组织、地点、物品、数字、时间、方向、距离和单位。
- 事件发生顺序、动作施受关系、因果、成功/失败结果和章末信息。
- 人物位置、伤势、资源、持有物、能力限制与已支付代价。
- 人物知道、怀疑、误信和不能知道的事实。
- 关系方向、潜台词、承诺、秘密和权力差。
- `RULE-*` 的条件、限制、成本、例外及 Canon 状态。
- 伏笔的读者可见信息、隐藏真义和当前揭示边界。
- 引号内原话、题记、刻意重复、仪式句、口头禅和作者指定保留句。
- 章节 frontmatter、标题、链接和可见性；除非本轮明确授权相应字段。

### 不得漂移的表达契约

- POV 人称、时态、叙述距离和自由间接引语边界。
- POV 人物的词汇、句法、感知过滤器、回避话题和情绪失控模式。
- 作品整体正式度、时代感、地域感、幽默浓度、暴力/情感尺度。
- 作者有意保留的含混、留白、重复、长句、碎句或陈词滥调式人物语言。

如果一句只有改变保护项才能变“更顺”，优先保真并标记为不改。具体化只能调用原文或已加载事实，不能凭空补天气、气味、身体反应或背景故事。

## 4. 先定问题性质与编辑强度

### 问题性质

把每项诊断分为：

- `correctness`：指代、语法、动作顺序、空间或 POV 明确错误。
- `comprehension`：读者难以确定谁做什么、重要信息被句法埋没。
- `voice_contract`：偏离已批准 POV、人物声音或作品语域。
- `rhythm_and_density`：句段速度、重复或修饰密度妨碍当前场景功能。
- `optional_taste`：存在多种有效写法，仅属偏好；默认不改，除非作者选择。

不要用“我觉得更好”作为理由。理由必须指向理解、节奏、声音、信息层级、动作可视性或作者目标。

### 编辑强度

- `minimal`：修正歧义、错漏、无意重复、明显修饰堆叠和轻微节奏问题；保持句序与用词主体。
- `standard`：允许拆并句、调整段内信息顺序、压缩解释、加强动作/对白衔接；不改变段落 beat、语气和信息量。
- `intensive-local`：仅对作者指定的弱段做局部重写，可重组段落内部表达，但必须保持相同 POV、事实、动机、动作结果和读者揭示。

强度是改动范围，不是问题严重度。问题多不自动等于重写；原稿 voice 鲜明时优先较轻强度。

## 5. 逐段诊断

使用临时键 `LINE-001`、`LINE-002`；不是永久实体 ID。每项至少写：位置、问题类型、触发文本、读者影响、建议动作、建议强度、保护项和是否属于可选偏好。

### 清晰度

- 指代是否有多个可能对象。
- 主语、动作、受力、结果和时间顺序是否可恢复。
- 长定语、插入语或被动结构是否隐藏重要动作。
- 含混是作者有意留白，还是信息组织失误。

### 具体性

- 抽象判断能否落回文中已经存在的动作、物件、选择或后果。
- “害怕、愤怒、危险、宏大”等结论是否重复解释已展示内容。
- 具体化是否会发明新事实；需要新增事实时停止，不补写。

### 句式与段落节奏

- 句长、停顿、段长是否服务行动速度、压力、犹豫或余波。
- 连续句是否同长度、同句式、同落点，产生机械抛光感。
- 碎句、长句、排比和反复是否属于人物/场景有意节奏。
- 高压动作段是否被解释拖慢，低速情绪段是否只剩静态抒情。

### 重复与修饰堆叠

- 区分信息重复、词汇近距重复、意象重复与有意母题。
- 同一句多个形容词、副词、比喻是否争夺焦点。
- 一个动作是否被动作、比喻和解释连续说三遍。
- 删除后若失去音韵、人物习惯或伏笔强调，则保留或只减一层。

### POV 与叙述距离

- 同一场景是否滑入非 POV 人物内心或使用 POV 不可能知道的信息。
- 词汇、判断和感官是否来自该人物，而非编辑者的通用文学腔。
- 贴身、近距、中距、远距切换是否有明确叙事目的。
- 是否为了悬念隐瞒 POV 此刻必然想到的内容。
- 时态和人称变化是否来自已批准结构，而非无意漂移。

### 对白区分

- 对白是否体现人物目标、策略、关系位置、知识和回避方式。
- 不同人物是否都使用相同的完整解释句、机智反击或总结腔。
- 称谓、礼貌程度、停顿和潜台词是否随对象变化。
- 动作节拍是否澄清说话者和空间，而不是每句都配表情。
- 不要靠方言拼写、口癖堆叠或故意语病制造差异。

### 动作逻辑

- 刺激、感知、反应、选择、动作和结果的顺序是否清楚。
- 身体姿态、手中物、距离、视线和受力能否同时成立。
- 人物是否瞬间完成多项互斥动作，或在未取得物品前使用它。
- 修句不能改变谁先动手、谁看见、谁受伤或行动是否成功。

### 感官选择

- 感官是否经过 POV 的注意过滤器，并影响当前判断或行动。
- 不机械补齐视觉、听觉、嗅觉、触觉、味觉“五件套”。
- 避免用通用气味、心跳、喉咙发紧等身体反应替代具体人物体验。
- 已有感官若承担空间、危险、记忆或伏笔功能，优先保留。

### 陈词滥调与 AI 套路

按上下文判断，不使用机械禁词表。重点识别：

- 通用警句、抽象拔高、无来源“有人说/人们都知道”。
- “不是 X，而是 Y”“不仅……更……”等伪洞见骨架。
- 戏剧化碎句、三件套、整齐排比、句长过匀和正能量收尾。
- narrator 反复解释“这意味着什么”“真正重要的是”，而动作已表达。
- 模型式开场、总结、替人物做心理诊断或替读者规定情绪。
- 过度平滑的因果连接、每段都先概括后说明、每句都显得会总结。
- 工程/商业/自媒体语域误入不相符的历史、人物或叙事场景。

保留条件：表达属于人物口吻、时代/职业术语、引用、反讽、仪式重复、主题母题或有意不可靠叙述。清理 AI 味不能把所有角色抹成同一种简洁 voice，也不能把小说改成说明文。

### Gate B：问题单与强度确认

先提交最重要的问题、保护项和推荐强度。作者可以接受、降低/提高强度、排除段落或把某写法标为有意。没有强度确认时不输出整章改写；若用户初始请求已明确强度与禁改项，回显诊断后可视为本 Gate 已满足。

## 6. 提供局部修订

对获准问题逐项输出：

```text
位置 / LINE-* / 强度
原文：保持原样引用
建议稿：只改必要句段
理由：问题如何影响理解、节奏、声音或动作
保留：列出未改变的事实、POV、语气和信息边界
```

默认每处只给一个推荐版本。只有两种写法会产生不同但都合理的语气/节奏时，才提供最多两个选项并说明取舍。不要借“备选”扩写出新情节。

局部修订原则：

- 优先删冗余、换清楚的主语/动作、调整停顿，再考虑重写。
- 保留作者原句中最有识别度的词、节奏和意象，不把所有句子平均抛光。
- 只修改已标问题的句段；无证据证明有问题的段落保持不动。
- 解释理由保持简短，不用一大段元话语证明编辑者会写。
- 对 optional taste 明确标“可不改”，让作者保留原文不承担隐性压力。

## 7. 三遍回读

### Pass 1：保真回读

逐项对照原稿：

1. 事实、数字、称谓、动作结果和责任主体是否漂移。
2. POV 知识、人物动机、关系和 Canon 是否改变。
3. 人物声音、叙述距离、时态和作品语域是否仍成立。
4. 信息是否丢失，留白或伏笔是否被提前解释。
5. 删并句后因果、空间和段落落点是否断裂。

任一项失败，退回原句或降低强度；不要用另一处补写掩盖漂移。

### Pass 2：残留模式审计

只在保真通过后检查：开场/收尾套话、narrator 解释、空泛判断、句长过匀、二元拔高、修饰堆叠和语域混搭。第二遍只允许轻量修正，不重新改写全文。

### Pass 3：朗读与角色区分

检查句子是否能自然朗读、停顿是否服务场景、连续对白能否不看名字仍感到人物差异。朗读顺不等于全部短句；角色可有笨拙、正式、绕弯或不完整的说法。

## 8. 作者接受门禁

在修改章节文件前交付：

- 问题单、编辑强度和保护项。
- 原文/建议稿对照及每项简短理由。
- `fidelity_check`：事实、POV、人物声音、动机、Canon 和揭示边界是否保持。
- `optional_choices`：仅属偏好的选择。
- `unresolved_upstream_issues`：需要 chapter/scene/character/continuity/Canon 流程的问题。

作者可逐项接受、拒绝、保留原文或要求另一强度。未明确接受时保持“建议稿”，不写回源文件。

作者明确授权写回后：

1. 用最小补丁修改唯一获准章节范围，不重排未授权段落。
2. 通过 Git diff 或等价差异视图保留原稿可比较；不清除用户已有差异。
3. 按实际正文更新 `word_count`；`content_version`、`status`、`updated_at` 仅按项目约定和作者授权调整。
4. 不修改 title、summary、章节链接、visibility 或其他 frontmatter，除非本轮明确授权且 Schema 合法。
5. 解析 Markdown/YAML frontmatter，并运行 `npm run validate:story`。
6. 再做一次保护项 diff 审计，确认没有新事实、情节、动机或 Canon 漂移。

作者接受 line edit 不代表章节自动 `locked/published`，也不批准写后提取的事实 proposal。

## 输出格式

每次响应至少包含：

- `novel_id`、`chapter_id`、卷章位置、精确范围和当前 Gate。
- 验证结果、上下文清单、保护项和编辑强度。
- `issues`：位置、类型、触发、影响、建议动作、是否可选。
- `local_revisions`：原文、建议稿、理由和保留项。
- `fidelity_check` 与 `residual_audit`。
- 作者已接受/拒绝/待定项，未修改的原稿范围。
- `author_acceptance_required`、精确 `stop_point`、下一步路由、写入文件与验证证据。

## 必须停止的情况

- 缺少明确 `novel_id`、`chapter_id`、唯一文件或精确章节范围。
- 注册、manifest、frontmatter 或场景 owner 不一致。
- 结构、人物或连续性 blocker 尚未解决。
- 修句必须改变事件结果、人物动机、知识、关系、规则、伏笔或 Canon 才能成立。
- 需要读取其他小说、未声明共享世界版本或无边界上下文来模仿风格。
- 目标是未授权的 `locked/published` 章节，或文件存在重叠用户改动。
- 作者尚未确认编辑强度，或尚未接受建议稿却要求静默覆盖原稿。
- 验证失败、保护项漂移或修改会扩大到未授权段落。

停止时给出文件、章节/场景 ID、具体证据、失败门禁和最小解阻动作。不要自行选择方便的版本。

## 完成检查

- 已显式绑定正确 `novel_id`、`chapter_id` 和授权范围。
- 已确认结构、人物与连续性门禁通过，并记录上下文预算。
- 已建立事实、POV、声音、动机、Canon 和揭示边界保护账本。
- 已先标问题与编辑强度，再给局部修订。
- 已检查清晰度、具体性、节奏、重复、修饰、POV、对白、动作、感官、陈词滥调和 AI 套路。
- 每项改动都有文本证据与功能理由，没有把个人偏好当规则。
- 原稿与建议稿可比较，未获接受的建议没有写回。
- 三遍回读确认没有情节、人物动机、关系、知识或 Canon 漂移。
- 获授权写回时只修改目标范围，并更新必要的合法 frontmatter。
- Markdown/YAML、`npm run validate:story` 和最终 diff 通过；已记录作者接受状态和精确停点，未提交 Git。
