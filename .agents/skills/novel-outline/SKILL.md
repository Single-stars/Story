# Novel Outline

把 premise 逐层扩张为可追踪的因果脊柱、人物弧与章节计划。把结构方法当作诊断镜头，不把百分比或固定节拍当作故事正确性的证明。

## 核心契约

- 在任何小说级读取或写入前要求用户明确给出并确认 `novel_id`。缺失时转交 `story-os-manager`，不要用 `active_novel_id` 或最近文件代替选择。
- 把版本化源文件视为唯一事实源。区分 Canon、proposal、research 和 manuscript；只允许作者批准 Canon。
- 只读取当前小说和 manifest 中显式声明、带版本的系列或共享世界依赖。不得借用其他小说的人物、设定或结构。
- 在读写前声明大纲层级、目标、上下文清单、上下文预算、修改文件和门禁。
- 把所有新大纲内容保持为 `proposed`。作者对某层的批准只允许继续下一层，不自动把新事实或大纲晋升为 Canon。
- 使用 `templates/narrative/outline.yaml` 创建正式提案；不要编辑模板本身。
- 把场景意图交给 `novel-scene`，把 premise 选择交给 `novel-premise`，把人物或世界新事实交给相应专用流程或 Canon 变更流程。

## 1. 绑定范围并先验证

1. 读取 `workspace/manifest.yaml`，验证用户给出的 `novel_id` 已注册且路径存在。
2. 读取目标小说 `manifest.yaml`，核对 `novel_id`、当前焦点、类型承诺、结构镜头、系列与共享世界版本。
3. 检查工作树是否有与目标大纲重叠的用户改动；存在重叠时停止并报告。
4. 从仓库根目录运行 `npm run validate:story`；在 Windows PowerShell 执行策略阻止脚本入口时使用 `npm.cmd run validate:story`。
5. 报告全部验证错误。Schema、归属、引用、共享世界版本或连续性错误会影响本次大纲时，停止写入；不要顺手修复。
6. 确认大纲层级：`novel`、`volume`、`act` 或 `chapter`，并确定唯一目标文件。

输出启动契约：`novel_id`、任务类型、层级、目标、输入状态、验证结果、拟读取文件、拟修改文件和第一个作者门禁。

## 2. 只加载相关上下文

默认只加载：

- `AGENTS.md`、目标小说 manifest 和已选定 premise。
- premise 直接依赖的类型承诺、主题问题与结局代价。
- 与当前范围有关的人物、规则、事实、关系、情节线、伏笔和作者决策。
- 修订任务中的现有大纲，以及与修改范围直接相邻的上层和下层结构。
- manifest 明确引用且当前大纲确实需要的系列或共享世界版本。

使用 `rg` 按 ID 和路径精确定位。维护上下文清单，列出已读取文件、事实 ID、依赖版本和明确排除的内容。默认不要读取其他小说、整部正文、根目录历史计划、构建物、依赖目录或日志。需要扩大预算时，先说明新增内容、原因和数量，再请求许可。

## 3. 固定输入契约

在展开结构前，用短表确认：

- premise 的一句话版本、状态与作者选择依据。
- 主角的外部目标、内部需要、创伤或错误信念。
- 核心对抗、失败代价、结局代价和主题问题。
- 主要类型、读者承诺、必要体验与不能误导的边界。
- 已批准的世界规则、时间限制、资源限制和能力代价。
- 本层大纲的起点状态、终点状态与成功标准。

不要自行补齐缺失的基础设定。把不阻塞的问题写入 `open_questions`；把会改变 premise、结局、世界规则或主要人物弧的问题标为阻塞，并停在作者门禁。

### Gate A：输入批准

要求作者确认范围、输入契约和阻塞问题的处理。未确认时只输出输入审计，不进入宏观结构设计。

## 4. 比较宏观结构方案

提供至少两个结构方案。每个方案都说明：

- 开局承诺如何兑现类型体验。
- 第一次重大转向、中段变化、最低点、高潮选择和结局代价。
- 核心读者体验、人物弧压力和信息揭示策略。
- 优势、代价、连续性风险、节奏风险和适用条件。
- 与 manifest 已声明结构镜头的关系，以及有意偏离的效果。

可以使用三幕、四幕、Snowflake、Story Circle、类型节拍或其他镜头进行比较，但不要把固定百分比、节拍名称数量或模板完成度当作质量标准。优先判断因果、选择、状态变化和读者承诺。

### Gate B：方案选择

要求作者选择、组合或拒绝方案。记录作者理由；未选择时不要替作者定案，也不要写入正式 outline 文件。

## 5. 建立因果脊柱

按所选方案填写 `causal_spine`。每个 beat 必须回答：

1. 哪个既有事件、事实或上一 beat 的后果触发当前压力？
2. 哪个角色想得到什么，动机来自 Want、Need、Wound、Lie、价值观或现实压力中的哪一项？
3. 阻力为何此刻生效并升级？
4. 角色作出什么选择或行动，而不是被巧合搬运到下一节点？
5. 转折改变了什么判断、资源、关系、风险或目标？
6. 行动造成什么可追踪的状态变化、事实或事件？
7. 该结果如何产生下一 beat 的具体压力？

用“因为 / 所以”朗读相邻 beat。若只能用“然后”，补齐原因、选择或后果。巧合可以制造麻烦，不能持续替角色解决麻烦。没有已分配的 `EVT-*` 或 `SCN-*` 时保持引用数组为空，不伪造 ID。

### Gate C：因果批准

向作者展示因果链、不可逆选择、最弱连接和至少一个删减判断。作者未确认因果脊柱前，不扩写人物弧账本或章节计划。

## 6. 对齐三本账

### 人物弧账本

对每名主要人物记录起始状态、Want、Need、Wound/Lie 压力、关键选择、不可逆选择、结束状态和关联 beat。检查人物变化是选择造成的，不是作者总结出来的。

### 情节线账本

对每条 `THREAD-*` 记录 dramatic question、开局状态、升级 beat、相互依赖、解决 beat 和预期解决方式。主线与支线每次交叉都必须改变资源、信息、关系或选择条件。

### 伏笔账本

对每条 `FORESH-*` 记录种植 beat、读者可见信息、真实含义、推进 beat、预计回收 beat 和失败风险。区分种植、推进、回收与放弃；放弃必须说明原因。避免只在回收前临时补种。

同时标出关系、知识、物品、伤势、时间和共享世界依赖的连续性风险。发现新事实需求时只创建提案，不静默写入 Canon。

### Gate D：账本批准

要求作者确认主要人物弧、情节线取舍、伏笔回收承诺和已知连续性风险。阻塞线未解决时不要进入章节拆分。

## 7. 拆分章节计划

为每章填写：

- 承接上一章的 `entry_state` 与 `causal_trigger`。
- 本章对主线、人物、关系、谜团、世界规则或类型体验的具体作用。
- 主导冲突、决定性变化和由结果产生的 `exit_hook`。
- 推进的人物、关系、情节线和伏笔 ID。
- 已存在的场景 ID；尚未经过场景设计时保持空数组。

每章至少推进一项核心承诺，重要章节应推进多项。合并功能重复的章节；删除后不影响因果、人物或读者承诺的章节视为弱章节。不要在大纲阶段写章节正文或把章节标题当作变化。

### Gate E：章节与写入批准

向作者提交章节计划、节奏风险、未决问题和拟写入路径。只有作者明确批准本层计划和目标文件后，才复制模板并写入提案。

## 8. 用 Scene/Sequel 与 Five Commandments 做兼容诊断

在大纲层只诊断，不生成正式场景契约：

- 对行动段检查 Goal / Conflict / Disaster；对反应段检查 Reaction / Dilemma / Decision。不要要求每章机械包含全部六项。
- 对关键宏观段落和高影响场景意图检查 Inciting Incident、Progressive Complication / Turning Point、Crisis、Climax、Resolution。
- 确保触发、危机、高潮行动和结果属于同一因果链，高潮来自人物选择。
- 确保每个重要结果能成为下一章或下一场的入口压力。

需要落成 POV、时间、地点、入场状态、目标、阻力、转折、危机、行动、结果、离场状态、知识变化、关系变化和伏笔引用时，转交 `novel-scene` 并使用 `templates/narrative/scene-contract.yaml`。

## 9. 写入 outline proposal

获得 Gate E 批准后：

1. 复制 `templates/narrative/outline.yaml` 到作者批准的目标小说叙事目录，不编辑模板本身。
2. 立即填写真实 `owner.novel_id`、层级、来源引用、依赖版本、目标、结构选择、因果脊柱、三本账、章节计划、风险和开放问题。
3. 保持 `status: proposed`、`visibility: internal` 和 `approval.status: pending`，除非存在可引用的作者决定记录；层级批准不等同于 Canon 批准。
4. 不伪造实体 ID、决策 ID、验证证据或作者批准时间。
5. 解析 YAML，并再次运行 `npm run validate:story`。记录命令、结果和仍存在的 warning；验证失败时不得宣称完成。
6. 检查 Git diff，确认只修改授权文件且没有跨小说泄漏。除非用户明确要求，否则不要提交 Git。

## 输出格式

每次响应至少包含：

- 绑定的 `novel_id`、大纲层级和当前 Gate。
- 验证结果与上下文清单。
- 使用的 Canon / proposal / research ID 及共享世界版本。
- 当前层提案、备选方案和明确取舍。
- 作者已确认决定与未批准内容。
- 风险、阻塞问题、精确停止点和下一步门禁。
- 写入文件与验证证据；尚未获写入批准时明确写“未落盘”。

## 必须停止的情况

- 缺少明确 `novel_id`，或注册表、路径、manifest、owner 不一致。
- premise 尚未由作者选择，或任务要求本 Skill 替作者批准基础方向。
- 验证失败会影响当前结构，或现有大纲有重叠的未提交用户改动。
- 需要读取未声明的其他小说或未锁定版本的共享世界。
- 必须新增或修改 Canon 才能继续，但没有作者批准的变更流程。
- 当前 Gate 尚未获得作者确认，或范围、结局、主要人物弧存在阻塞歧义。
- 请求开始写正文、越过章节范围、静默创建场景契约或直接修改共享世界。

停止时报告具体文件、ID、失败门禁和最小解阻动作；不要自行选择最方便的故事版本。
