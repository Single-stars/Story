# Novel Canon Change

把本 Skill 当作 Canon 变更协调器和审批门禁。先证明变更范围与代价，再等待作者决定；不要把“更合理”当成批准。

## 核心契约

- 在任何小说级读取或写入前，要求用户明确给出并确认 `novel_id`；不得按最近文件、唯一候选或 `active_novel_id` 猜测。
- 把版本化源文件视为唯一事实源，严格区分 Canon、proposal、research、manuscript 和派生物。
- 作者是 Canon 的唯一批准者。AI、研究、反馈、审查和正文提取只能发起 `idea`、`proposed`、audit 或 `CR-*`。
- 未取得针对具体 `CR-*` 的作者批准证据前，不修改现有 Canon 状态、事实、引用或正文；变更请求保持 `status: proposed`，`approval.status: pending`，`application.applied: false`。
- 保留旧事实、来源、原因、替代方案、影响范围和替代事实。不得删除、覆盖历史或复用稳定 ID。
- 发现冲突时并列证据和选择，不替作者决定哪个版本为真。
- 小说本地 Canon 不得覆盖 `UNIV-*` 所有的实体。共享世界变更必须转到对应 universe owner，并在共享世界范围审计所有受影响作品和引用版本。
- 预计修改超过 3 个文件时拆成小批次，每批声明范围、验证并等待下一步授权。

## 1. 绑定变更范围

1. 按 `story-os-manager` 核对工作区注册、小说目录和 manifest 的 `novel_id` 一致。
2. 声明任务类型为 Canon change，并确认意图：`add`、`modify`、`deprecate` 或 `retcon`。
3. 列出目标实体 ID、当前状态、正式所有者、源文件和请求来源。
4. 确认基准提交或当前工作树；保留用户未提交内容，不混用不同历史版本。
5. 声明本轮只做提案、影响审计、审批记录或已批准变更应用中的哪一步。
6. 运行 `npm run validate:story`；Windows PowerShell 入口受限时使用 `npm.cmd run validate:story`。

启动输出至少包含：`novel_id`、变更意图、目标 ID、owner、基准、当前 Canon 状态、读取范围、修改范围、验证结果、作者待决问题和停止 Gate。

### Gate A：身份、归属与基准

缺少明确 `novel_id`、目标 ID、正式 owner 或一致基准时停止。目标由 `universe_id` 所有时，不进入小说本地应用流程，转到“共享世界边界”。

## 2. 建立最小上下文

默认只读取：

- `AGENTS.md`、`README.md`、`STORY_OS_SPEC.md` 的 Canon/ownership/validation 条款。
- `workspace/manifest.yaml`、目标小说 manifest 和相关共享世界版本声明。
- 目标 Canon 源文件、直接引用它的实体和最近相关作者决策。
- 受影响的事件、场景、章节、人物知识、关系、物品、伏笔和线索记录。

先用 `rg` 按实体 ID 和路径精确搜索。默认最多加载 8 个源文件或 20 个直接相关实体；不足时列出下一依赖环、理由和预计数量再扩展。不得把无关小说、读者反馈或研究推断当成正式事实。

记录 `context_inventory`：已读文件、事实 ID、状态、共享世界版本、排除范围和覆盖缺口。

## 3. 判断变更等级

以下任一情况按高影响变更处理并必须使用 `CR-*`：

- 替换、否定或追溯修改现有 Canon。
- 改变世界规则、人物历史/身份/能力、关键事件因果或结局条件。
- 影响已写正文、时间线、人物知识、关系有效区间、关键物品或伏笔公平性。
- 影响系列、共享世界、阅读顺序、可见性或已发布内容。
- 需要迁移引用、重写多个文件或具备非平凡回滚成本。

低影响新增也先形成结构化 proposal；除非仓库协议明确允许，否则不要绕过作者审批直接晋升 Canon。无法判断等级时按高影响处理。

## 4. 创建完整变更请求

复制 `templates/governance/change-request.yaml` 到当前小说约定的正式位置后填写，不要修改模板本身。分配未占用的 `CR-*`，并确保 `owner.novel_id` 等于当前 `novel_id`。

提交审批前逐项补全：

1. **身份与请求：**`id`、owner、标题、摘要、请求者、时间、可见性和 change type。
2. **旧事实：**当前实体 ID、Canon 状态、精确陈述、源文件/字段/行或提交引用；不得用概括抹掉旧版本。
3. **新事实：**候选实体 ID、完整陈述、目标状态、`supersedes_ids`、`replacement_ids` 和显式假设。
4. **原因：**要解决的真实写作或连续性问题、为何现在变更，以及“不变更”的后果。
5. **替代方案：**至少包含 `keep_current` 和一个较小改动方案；记录收益、成本与拒绝理由，未决定前不得预填作者选择。
6. **影响 ID：**Canon、事件、场景、线索、伏笔、知识、关系、物品、决定、小说、系列和共享世界 ID；无影响也显式写空数组。
7. **受影响正文：**逐一列出 manuscript path、章节/段落锚点、当前表述、所需处理与读者影响；正文不能只写“全书搜索”。
8. **时间影响：**故事时间、叙事顺序、因果边、人物位置、移动/恢复耗时和事件精度。
9. **知识影响：**谁在何时通过什么来源知道/怀疑/误信/验证哪些 `FACT-*`，以及 `knowledge_used_ids` 是否需迁移。
10. **关系影响：**分别审计关系方向、有效区间、原因事件、公开标签、私下现实与受影响场景。
11. **物品与资源：**持有人、地点、状态、转移事件、消耗与后续使用是否仍成立。
12. **伏笔与线索：**种植、推进、回收/放弃、真实含义、预期回收范围和剧透边界。
13. **分享影响：**最高审查 profile、已发布内容、摘要/索引/reader bundle 是否会泄漏或失真。
14. **迁移：**前置条件、按序步骤、目标路径、引用更新、兼容说明和每步预期结果。
15. **验证：**解析、Schema、ID/owner/reference、状态迁移、时间/知识/关系/物品/伏笔、正文搜索、可见性构建，以及对应命令和预期结果。
16. **回滚：**触发条件、可恢复的旧引用/提交、逆序步骤、回滚后验证命令和证据保存位置。

任何必填分析未知时写入 `assumptions` 或 coverage gap，并阻止审批就绪；不要用空字符串伪装完成。

### Gate B：CR 完整性

只有旧事实、新事实、原因、替代方案、全部影响面、迁移、验证和回滚均可执行时，才把请求标记为 `ready_for_author_decision`。这只是工作流说明，不得把 CR 的 `status` 改为 `canon`。

## 5. 做影响审计

从目标 ID 向外追踪引用，不从“可能相关”开始无边界扫描：

1. 建立 `current_state -> proposed_state` 差异表。
2. 按事件因果和 `story_time` 检查时间链；另按场景/章节检查 `narrative_order`。
3. 对每个受影响人物建立变更前后知识、位置、伤势、资源和行动依据。
4. 对每条 `REL-*` 分方向检查有效区间和原因事件。
5. 对每个 `ITEM-*` 检查 `item_changes` 的前后持有人、地点和状态。
6. 对每个 `FORESH-*` 检查 planted/advanced/paid_off/abandoned 及读者公平性。
7. 精确搜索正文、摘要、索引、场景契约、outline、报告和 reader/public 源数据中的引用。
8. 区分必须迁移、建议修订、无需改变和证据不足四类结果。

审计报告必须引用路径、字段/锚点、实体 ID、观察状态和预期约束。验证器未报错不等于影响审计完整。

## 6. 共享世界边界

若目标实体由 `universe_id` 所有：

1. 停止创建或应用小说本地替代实体；不得用相同 ID、复制事实或本地例外覆盖共享世界。
2. 把请求 owner 改为对应 `UNIV-*`，转到该共享世界约定的变更位置和所有者范围。
3. 记录当前 `novel_id` 作为受影响项目，而非正式 owner。
4. 审计所有引用该共享世界版本的小说、系列、正文、时间、知识、关系、伏笔、可见性与版本兼容性。
5. 给出共享世界版本升级、小说继续固定旧版本、兼容层或拒绝变更等方案。
6. 等待共享世界所有者/作者批准；批准前当前小说继续读取已声明版本。

缺少 UNIV owner、版本或跨作品审计权限时停止，只保留 proposed CR 和依赖缺口。

## 7. 作者审批 Gate

向作者交付精简决策包：`CR-*`、旧事实、新事实、原因、替代方案、主要影响、迁移成本、验证、回滚和必须选择的问题。

只有同时满足以下条件才视为批准：

- 作者明确点名具体 `CR-*` 和批准的方案/最终事实。
- 批准范围、例外、迁移时机和目标状态无歧义。
- 创建 `DEC-*` 作者决策记录，并在 CR 中填写 `decision_record_id`、`decided_by`、`decided_at` 和 rationale。
- 决策记录与当前基准、owner 和 CR 内容一致；CR 在等待期间发生实质变化时必须重新批准。

处理结果：

- **批准：**`approval.status: approved`，进入应用计划；批准本身不等于已应用。
- **拒绝：**保留 CR、旧事实和拒绝理由，标记 rejected，不改 Canon。
- **要求修改/部分批准/含糊回应：**CR 继续保持 proposed/pending，列出待确认项并停止。

不得代填作者姓名、批准时间、签名、Git 身份或决定理由。

### Gate C：应用授权

即使已有批准，仍要复述将修改的文件和批次。若批准未覆盖正文、共享世界或发布数据，不得顺带修改这些层。

## 8. 应用已批准变更

按批准的迁移步骤小批执行：

1. 重新验证基准和 CR 未过期；若目标文件已变化，停止并重新做影响审计。
2. 先保存决策证据，再修改 Canon。
3. 对 retcon，把旧记录标为 `retconned` 并写 `superseded_by`；让新记录使用自己的稳定 ID 和批准状态。
4. 对 deprecate，保留旧记录与原因，不伪造替代事实。
5. 对 modify/add，保持历史可追踪；不得通过改名复用已删除或旧事实 ID。
6. 更新批准范围内的引用、事件、知识、关系、物品、伏笔和时间记录。
7. 正文修订交给 `novel-chapter`，场景交给 `novel-scene`，结构交给 `novel-outline`；逐文件列出授权，不把 Canon 应用自动扩张为全书改写。
8. 每批运行计划中的验证；失败立即停止后续批次，保留证据并判断修复还是回滚。
9. 全部成功后填写 `application.applied`、时间、执行者、source commit（若存在）、resulting IDs 和 verification evidence。

没有作者授权的依赖修订保持 proposal 或待办，不得为了通过验证偷偷改写。

## 9. 验证与回滚

至少执行：

- YAML/Markdown 解析和适用 Schema 校验。
- `npm run validate:story` 或 Windows 下 `npm.cmd run validate:story`。
- ID 唯一、前缀、owner、引用、共享世界版本和 Canon 状态迁移检查。
- 时间、人物位置、知识 learned/used、关系有效区间、`item_changes` 和伏笔回收复核。
- `rg` 搜索旧陈述、旧 ID、受影响正文和遗漏引用。
- 适用时执行类型检查、测试、reader 数据构建和目标 profile 隐私检查。

把命令、退出码、结果摘要和证据路径写回 CR。验证失败时不得宣称应用完成、构建发布数据或提交。

触发回滚时按 CR 中步骤逆序执行，恢复旧引用和状态，重新运行同一组验证，并记录回滚原因、执行范围和遗留问题。回滚不能删除 CR、DEC 或旧事实历史。

## 10. 输出与停点

按当前阶段输出：

1. `change_header`：novel_id、CR、owner、基准、阶段和文件范围。
2. `current_vs_proposed`：旧事实、新事实、原因和替代方案。
3. `impact_matrix`：ID、正文、时间、知识、关系、物品、伏笔、分享和共享世界影响。
4. `migration_validation_rollback`：可执行步骤、命令、预期和触发条件。
5. `approval_state`：pending/approved/rejected、DEC 证据及不可执行范围。
6. `application_state`：未应用、部分应用、已应用或已回滚及验证证据。
7. `stop_point`：完成位置、未决问题、下一批文件、下一步路由和最小解阻动作。

## 必须停止的情况

- 缺少明确 `novel_id`，或注册、manifest、owner、路径、版本基准不一致。
- 旧事实或新事实无法精确陈述，影响 ID/正文/时间/知识/关系/物品/伏笔尚未审计。
- 请求删除历史、复用稳定 ID、把 research/feedback/manuscript 推断直接写成 Canon。
- 作者未明确批准具体 CR，批准证据缺失、含糊、过期或与当前 CR 不一致。
- 小说本地请求试图覆盖共享世界实体，或缺少 UNIV owner/版本/跨作品权限。
- 迁移、验证或回滚不可执行，上下文预算需要无边界扩张，或文件改动与用户工作重叠。
- 任一验证失败、可见性可能泄漏，或需要修改未获授权的正文/发布数据。

停止时保留 CR 为 proposed，给出具体文件、ID、失败 Gate、已完成分析和最小解阻动作。

## 完成检查

- 已显式绑定 `novel_id`、目标 ID、owner、基准和变更阶段。
- CR 已完整记录旧/新事实、原因、替代方案、影响 ID 和正文位置。
- 已审计时间、知识、关系、物品、伏笔、分享与共享世界版本。
- 迁移、验证、回滚均有顺序、目标、预期结果和证据要求。
- 未批准内容仍为 proposed；Canon 只在具体作者决定记录后变化。
- retcon 保留旧事实并记录 `superseded_by`，稳定 ID 未被复用。
- 共享世界变更已转到 UNIV owner，没有小说本地覆盖。
- 已运行计划内验证并记录结果，失败时未继续发布或提交。
- 已检查 Git diff，只修改授权文件，记录精确停点且未伪造 Git 身份。
